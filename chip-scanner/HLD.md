# 筹码扫描系统 高层架构设计 (HLD)

> AKShare 指标粗筛 + PC 端视觉终审 的"漏斗模型"量化筛选系统
> 目标:最大化节省 AI API 成本与本地算力,自动捕获"低位单峰套牢盘"种子股。

---

## 1. 系统整体架构 (System Architecture)

五层结构:数据层 → 任务调度层 → 策略过滤层 → 视觉捕获层 → 通知层。

```
                    +------------------------+
                    | 定时任务调度 (Cron/APS) |
                    +------------+-----------+
                                 |
                                 v
                     +----------------------+
                     | 数据抓取层 (AKShare)  |
                     +-----------+----------+
                                 |
                                 v
+------------------------------------------------------------------+
|                     策略过滤层 (漏斗过滤器)                       |
|                                                                  |
|  [周级/两周] 冰封池  -->  [每2天] 观察池   -->  [每日盘后] 临界池   |
|  (全市场/基础垃圾过滤)    (筹码集中度 SCR 改善)   (符合单峰数值临界) |
+------------------------------------------------------------------+
                                 |
                                 | (仅输出 10-20 只高价值种子股)
                                 v
                  +------------------------------+
                  |  视觉捕获与 AI 终审层        |
                  |  - PyAutoGUI (控制东吴秀财)  |
                  |  - 多模态 AI / OpenCV 核心   |
                  +--------------+---------------+
                                 |
                                 v
                    +------------------------+
                    |    通知与报警模块       |
                    |  (微信/钉钉/本地日志)   |
                    +------------------------+
```

## 2. 核心模块详细设计

### 2.1 数据抓取与存储层 (Data Layer)
- **职责**:获取 A 股全市场基本面、行情、筹码衍生数据。
- **技术选型**:Python AKShare(抓取) + 本地 SQLite(股票池状态与监控级别)。
- **关键数据项**:股票代码、收盘价、换手率、历史筹码分布获利盘比例、90% 筹码集中度 (SCR)。

### 2.1.1 基础过滤硬规则 (Universe Filter — 冰封池入口)
全市场进入系统前先做硬性剔除,缩小候选池:

| 规则 | 条件 | 实现要点 |
|---|---|---|
| 排除科创板 | 代码 `688xxx` | 前缀过滤 |
| 排除创业板 | 代码 `300xxx` / `301xxx` | 前缀过滤(权限缺失) |
| 排除 ST/*ST | 名称含 `ST` / `*ST` / `退` | 名称匹配 |
| 排除高价股 | 收盘价 > ¥1000 | 价格过滤 |
| **流动性及格线** | 日成交额 < ¥5000 万 → 剔除 | 绝对流动性下限 |

> 保留范围:沪市主板 `60xxxx`、深市主板 `00xxxx`、北交所视需求另议。

### 2.1.2 流动性健康度评估 (Liquidity Health — 量化打分维度)
成交额"及格"后,再用"成交额 ÷ 流通市值"判断交投热度是否健康。

**绝对成交额四大生命线**
| 区间 | 状态 | 处理 |
|---|---|---|
| < ¥2000 万(尤其 <¥1000 万) | 僵尸股/仙股化,流动性枯竭 | 直接淘汰 |
| ¥2000 万 ~ ¥5000 万 | 缺乏关注区,流动性折价 | 淘汰(及格线 5000 万) |
| ¥1 亿 ~ ¥5 亿(1.5~3 亿最佳) | 中坚力量区,买卖滑点小 | 健康,优先 |
| ≥ ¥10 亿 | 权重股 / 风口题材龙头,博弈剧烈 | 需结合形态另判 |

**动态比例尺(核心)**:健康日成交额 ≈ 流通市值的 **1.5% ~ 3%**。
| 类型 | 典型流通市值 | 健康成交额区间 |
|---|---|---|
| 小/微盘 | 30 亿 | 4500 万 ~ 1 亿(<3000 万危险) |
| 中盘骨干 | 150 亿 | 2.25 亿 ~ 4.5 亿(机构建仓最舒服) |
| 大盘权重 | 1000 亿 | 10 亿 ~ 20 亿 |
| 超级巨无霸 | 5000 亿+ | 20 亿 ~ 50 亿 |

**两类成交额异动(降级/警示标记,不直接出信号)**
- **缩量空涨(无量空涨)**:连续涨但成交额极低 → 控盘度高/惜售,开板易踩踏。
- **爆量滞涨(巨量死亡震荡)**:成交额突然放大数倍但收长上影/涨幅微小 → 高位出货,极不健康。

> 算法落地:`换手率比例 = 成交额 / 流通市值`,落在 1.5%~3% 记健康分;偏离 + 价格异动组合判定上述两类异动。

### 2.2 状态机与策略过滤层 (Strategy & Funnel Layer)
系统维护股票状态表,每只股票存在三种状态,按周期流转:

| 监控级别 (Level) | 执行周期 | 触发条件 / 过滤算法 | 状态去向 |
|---|---|---|---|
| **Low (冰封池)** | 每周五盘后 | 剔除 ST、市值过大/过小、近期无成交量死股 | 达标→Mid,否则留守 |
| **Mid (观察池)** | 每 2 天盘后 | 近期筹码集中度变动;SCR < 10%(筹码高度密集)且股价在密集区下沿 | 达标→High,退化→Low |
| **High (临界池)** | 每日盘后 | 数量通常 ≤ 20 只,直接输出给下游 | 不达标则降级 |

### 2.3 视觉捕获与 AI 辨识层 (Vision & AI Layer)
连接"量化数据"与图形形态判定的桥梁,系统核心创新点。

> **⭐ 重大架构简化 (已验证)**: 我们的 SCR 算法内部已持有完整 `价格→筹码量` 分布,
> 可用 matplotlib **自绘标准筹码分布图** (`src/plot_chips.py`), **无需东吴秀财截图**。
> 这省掉了 PyAutoGUI / 坐标漂移 / 弹窗容错 / 渲染等待 等全部脆弱环节。
> 自绘图直接喂给 AI/OpenCV 判形态。**方案 A (自绘) 优先, 方案 B (券商截图) 仅作交叉校验备用。**

**方案 A — 自绘筹码图 (主, 已实现)**
1. 读取 High 池代码。
2. `compute_distribution()` 输出 grid+chips 分布。
3. matplotlib 绘横向筹码图: 红=获利盘 / 蓝=套牢盘 + 现价/平均成本/90%区间横线。
4. 输出 `output/charts/{code}.png`。

**方案 B — 券商截图 (备用, 待装东吴秀财)**
1. PyAutoGUI 聚焦窗口 → 输入代码 → `locateOnScreen` 特征图标相对定位 → ROI 截图。

**图形决策核心 (AI / OpenCV)**
- **API 优先**: 图片 (自绘 or 截图) 喂多模态大模型, 结构化 Prompt 返回 0/1 + 单峰/多峰判定。
- **本地轻量**: OpenCV 颜色通道识别红/蓝比例 + 峰型检测 (单峰 vs 多峰)。

> 验证发现: 纯 SCR 数值最优者 (如 002445 中南文化 SCR=0.108) 形态上可能是轻微双峰,
> 故形态终审 (区分单峰/多峰) 确有必要——这正是视觉层的价值。


### 2.4 任务调度与通知层 (Orchestration & Notification)
- **调度器**:APScheduler 或本地 Cron。
  - 每日 15:30 触发 Mid + High 任务。
  - 每周五 16:00 触发 Low 全市场大清洗。
- **通知器**:AI 终审确认"完美低位单峰套牢盘"→ 触发 PushDeer / 企业微信机器人 / 钉钉 Webhook,推送代码 + 现价 + 筹码截图。

## 3. 核心数据流向 (以股票 X 为例)
1. **周五晚**:Data Layer 抓全市场 → Funnel 发现 X 换手率抬升、形态初显 → `X.level = Mid`。
2. **周二晚**:调度器执行 Mid → X 筹码集中度跌破 12% → 晋升 → `X.level = High`。
3. **周三 15:30**:执行 High → 驱动东吴秀财切到 X → 截图 `X_chips.png` → AI 确认 90%+ 套牢盘且单峰完美 → 报警 → 手机收到带图推送。

## 4. 关键设计考量与容错 (Trade-offs & Resilience)
- **UI 阻断容错**:PC 软件偶发"版本更新/广告/断网"弹窗。对策:截图前用 pygetwindow 检测窗口是否前台;设计固定"关闭弹窗"动作,异常时报错并终止。
- **风控与频控**:大模型 API 有成本与并发限制。对策:严格漏斗保证每日 High 池 ≤ 20 只,一天几毛钱可接受。
- **坐标漂移**:分辨率改变或窗口拖动导致绝对坐标失效。对策:用 `pyautogui.locateOnScreen()` 定位特征图标(如"筹码"按钮)作基准,相对偏移截图。

### SCR 说明
SCR = Share Concentration Ratio (筹码集中度) = 东财"90集中度" = `(90成本高 − 90成本低)/(90成本高 + 90成本低)`。越小越密集(单峰)。

### ⭐ 限流终极解法: 状态机 + 预算上限
**问题**: 全市场 1897 只逐只拉筹码必被东财封 IP (实测跑到 ~1600 只触发, 累积越久越顽固)。
**解法**: 三级漏斗 + Mid 池预算上限 `MID_BUDGET=300`:
- **Low 池** (~1597): 仅用一次批量快照流转, **不逐只算筹码** (零额外请求)。
- **Low→Mid 晋升**: 廉价信号 (60日跌幅 ≤ -5% + 流动性健康), 按跌幅排名补位到预算上限。
- **只对 Mid+High (≤300 只) 算筹码** → 每日筹码请求从 1897 降至 ≤300, **永不触发封禁**。

### 晋升门槛 (集中配置在 orchestrator.py)
> ⚠️ **2026-06-15 最终定稿 (经用户逐图校准)**: 策略 = "**盈利 + 单峰密集**"。
> 经多轮人工核对图形, 确立**双指标判据**: 单峰(次峰比) + 密集(带宽70)。

| 流转 | 信号 | 阈值/逻辑 |
|---|---|---|
| **universe** | 基础硬过滤 + 剔除亏损 | PE_ttm > 0 |
| **Low→Mid** | 横盘整理 + PE>0 (仅快照) | `|60日涨跌幅| ≤ 30%`, 升序补位到 ≤300 |
| **Mid→High** | **单峰密集三重判据** (120日窗口) | ① 单峰: 次峰/主峰高度比 ≤ 0.5; ② 密集: 带宽70 ≤ 9%; ③ 非套牢: 现价相对主峰 ≥ -5% |
| **High 排序** | 带宽70 升序 | 越密集越靠前, 取 Top-20 |

**核心指标** (chip_calc.py, 120日窗口):
- `次峰比` = 平滑后次峰/主峰高度比 (**单峰判据**; 0=纯单峰, 接近1=真双峰)。**这是区分单峰vs多峰最准的指标**。
- `带宽70` = 70% 筹码价格带宽 ÷ 现价 (**密集判据**; 越小越密集)
- `距主峰` = (现价−主峰)/主峰 带符号 (>0 突破/启动, <0 套牢压顶)
- `尖锐度`(±2%) / `主峰占比`(±6%) 保留作辅助参考

> **校准历程 (用户逐图否定推动)**:
> 1. SCR90→SCR70→尖锐度(±2%): 越来越聚焦近期密集峰。
> 2. K线 250→120日: 250日把旧分散筹码画进来稀释峰, 导致"图看着不尖"。
> 3. 加抛弃套牢盘 (距主峰带符号): 排除常熟银行(现价低于峰10%)。
> 4. 加单峰纯度→峰度比→**最终次峰比**: 区分江苏金租(矮胖丘)/北京银行(双峰)。
> 5. 带宽放宽到9%: 用户认可盛路通信(带宽8.4%单峰)。
> **关键教训**: 固定窗口占比(主峰占比±6%)会把相邻多峰罩进去误判单峰; 必须用峰检测算次峰比。

### 关键技术决策 / 踩坑
1. **数据源限流**: 东财 push2/push2his 高频封 IP。改腾讯 `proxy.finance.qq.com` 日K (含换手率), 300只0失败。
2. **状态机 + 预算上限**: SQLite, Mid 池 ≤300, 每日筹码请求 ≤300 不触发限流。
3. **纯 Python 三角分布筹码算法**: 替代 akshare py_mini_racer。但日K天然偏宽, 做不出极尖塔。
4. **自绘筹码图 + 次峰比峰检测**: 替代券商截图; 峰检测是单峰判据核心。
5. **次新股剔除 + 剔除亏损 (PE≤0)** + **流通市值不设区间**。
